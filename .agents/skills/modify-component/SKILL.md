# 기존 컴포넌트 수정 워크플로우

기존 컴포넌트(Atom / Molecule / Organism)를 안전하게 수정하는 워크플로우.
새 컴포넌트 생성이 아닌 **기존 파일 변경**에 특화되어 있다.

## 워크플로우

### Step 1: 영향 분석 (읽기 전용)

1. 대상 파일 + 테스트 파일 읽기 (없으면 Step 2에서 신규 작성)
2. grep으로 사용처 탐색 → Props 변경 시 파급 범위 파악
3. 기존 테스트가 검증하는 동작 파악

### Step 2: 테스트 업데이트 (test-writer 에이전트)

- 새 동작 → 새 테스트 추가 (Red)
- 기존 동작 변경 → 해당 테스트 업데이트
- 삭제된 동작 → 해당 테스트 제거
- Props 제거 시 사용처 테스트도 함께 업데이트

### Step 3: 컴포넌트 수정

계층에 따라 에이전트 선택: `ui/atoms/`, `ui/molecules/` → `atom-implementer` / `editor/`, `pdf/` → `editor-implementer`

- 요청된 변경만 수행, 범위 밖 리팩터 금지
- `any`, `as`, `!` 금지

### Step 4: 사용처 업데이트 (Props 추가/제거/이름 변경 시)

영향받는 파일 순서대로 수정, `pnpm tsc --noEmit`으로 타입 에러 0 확인

### Step 5: 검증

`/verify` 스킬 호출 (`test → tsc → lint → build`)

## 유형별 체크리스트

### Props 추가
- [ ] 새 Props 인터페이스에 추가
- [ ] 필수값이면 사용처 전부 업데이트
- [ ] 선택값(`?`)이면 기본값 또는 undefined 처리
- [ ] 테스트에 새 prop 케이스 추가

### Props 제거 / 이름 변경
- [ ] 사용처 grep으로 전부 찾음
- [ ] 사용처 일괄 수정 완료
- [ ] 관련 테스트 업데이트

### 로직 변경 / 버그 수정
- [ ] 버그를 재현하는 실패 테스트 먼저 작성 (Red)
- [ ] 수정 후 테스트 통과 (Green)
- [ ] 기존 테스트 깨지지 않음

### 스타일 변경
- [ ] Tailwind 클래스만 사용 (커스텀 CSS 금지)
- [ ] 스타일 관련 테스트 업데이트 (클래스명 검증 테스트가 있으면)

### 리팩터
- [ ] 동작 변경 없음 (테스트 모두 그대로 통과)
- [ ] 200줄 초과 시 분리
- [ ] useCallback / useMemo 누락 없음

## 완료 기준

- [ ] 수정 대상 파일의 테스트 존재 및 통과
- [ ] `/verify` 스킬 전체 통과 (test → tsc → lint → build)
