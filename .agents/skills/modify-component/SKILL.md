# 기존 컴포넌트 수정 워크플로우

기존 컴포넌트(Atom / Molecule / Organism)를 안전하게 수정하는 워크플로우.
새 컴포넌트 생성이 아닌 **기존 파일 변경**에 특화되어 있다.

## 트리거 조건

- "[컴포넌트명] 수정해", "이 컴포넌트에 [기능] 추가해"
- "버그 고쳐줘", "[컴포넌트명] 동작이 이상해"
- "[컴포넌트명] 리팩터해줘", "코드 정리해줘"
- Props 추가/제거, 스타일 변경, 로직 수정

## 워크플로우

### Step 1: 영향 분석 (읽기 전용)

수정 전 반드시 현재 상태를 파악한다:

```
1. 대상 컴포넌트 파일 읽기
2. 대응하는 테스트 파일 읽기 (없으면 Step 2에서 신규 작성)
3. 대상 컴포넌트를 import하는 파일 확인 (grep으로 사용처 탐색)
4. 변경이 Atom→Molecule→Organism 계층에 미치는 파급 범위 파악
5. Props 변경 시 사용처 전체 목록 정리
```

**확인 포인트:**
- 변경하려는 Props/동작이 몇 곳에서 사용되는가?
- 기존 테스트가 어떤 동작을 검증하고 있는가?
- 변경이 기존 테스트를 깨뜨리는가?

### Step 2: 테스트 업데이트 (test-writer 에이전트)

```
test-writer 에이전트를 사용해서:
1. 기존 테스트 파일을 읽고 현재 커버리지 파악
2. 수정 사항에 맞춰 테스트 추가/수정
   - 새 동작 → 새 테스트 추가 (Red)
   - 기존 동작 변경 → 해당 테스트 업데이트
   - 삭제된 동작 → 해당 테스트 제거
3. pnpm test → 신규/변경 테스트가 Red 상태인지 확인
```

**주의:**
- 기존에 통과하던 테스트가 실패하면 → 의도된 파괴적 변경인지 확인
- Props 제거 시 사용처 테스트도 함께 업데이트

### Step 3: 컴포넌트 수정 (구현 에이전트 선택)

대상 컴포넌트 계층에 따라 에이전트 선택:

| 위치 | 에이전트 |
|------|---------|
| `ui/atoms/`, `ui/molecules/` | `atom-implementer` |
| `editor/`, `pdf/` | `editor-implementer` |

```
선택한 에이전트를 사용해서:
1. 대상 파일만 최소한으로 수정 (관련 없는 코드 건드리지 않음)
2. Props 변경 시 사용처(import한 파일들) 일괄 업데이트
```

**수정 원칙:**
- 요청된 변경만 수행, 범위 밖 리팩터 금지
- 200줄 초과 시 분리 검토 (단, 요청 범위 내에서만)
- `any`, `as`, `!` 금지 — 타입 에러는 근본 원인 수정

### Step 4: 사용처 업데이트 (필요 시)

Props 추가(필수값)/제거/이름 변경이 있는 경우:

```
1. 영향받는 상위 컴포넌트 파일들 순서대로 수정
2. 각 수정 후 pnpm tsc --noEmit로 타입 오류 확인
3. 타입 에러가 0이 될 때까지 반복
```

### Step 5: 검증 (/verify 스킬)

`/verify` 스킬을 호출해서 전체 검증을 실행한다.

```
test → tsc → lint → build 순서로 통과 확인
```

## 유형별 체크리스트

### Props 추가
- [ ] 새 Props 인터페이스에 추가
- [ ] 필수값이면 사용처 전부 업데이트
- [ ] 선택값(`?`)이면 기본값 또는 undefined 처리
- [ ] 테스트에 새 prop 케이스 추가

### Props 제거 / 이름 변경
- [ ] 사용처 grep으로 전부 찾음
- [ ] 사용처 일괄 수정 완료
- [ ] 관련 테스트 업데이트

### 로직 변경 / 버그 수정
- [ ] 버그를 재현하는 실패 테스트 먼저 작성 (Red)
- [ ] 수정 후 테스트 통과 (Green)
- [ ] 기존 테스트 깨지지 않음

### 스타일 변경
- [ ] Tailwind 클래스만 사용 (커스텀 CSS 금지)
- [ ] 스타일 관련 테스트 업데이트 (클래스명 검증 테스트가 있으면)

### 리팩터
- [ ] 동작 변경 없음 (테스트 모두 그대로 통과)
- [ ] 200줄 초과 시 분리
- [ ] useCallback / useMemo 누락 없음

## 완료 기준

- [ ] 수정 대상 파일의 테스트 존재 및 통과
- [ ] `/verify` 스킬 전체 통과 (test → tsc → lint → build)
